# Sales Forecasting. CatBoost vs Linear Regression

Прогнозирование недельных продаж на реальном датасете Walmart Recruiting -
Store Sales Forecasting (Kaggle). Данные на уровне Store-Dept: 45 магазинов,
81 отдел, ~420 тысяч наблюдений за 2010-2012.

## Что внутри

- `data_processor.py` - загрузка train/features/stores, объединение, очистка,
  генерация лагов и скользящих средних по каждой паре Store-Dept
- `train_predict.py` - обучение моделей, разбиение по времени, метрики, график
- `eda_notebook.ipynb` - разведочный анализ: сезонность, эффект праздников,
  разброс между типами магазинов и отделами, пропуски в маркдаунах

Данные лежат в `data/` (train.csv, features.csv, stores.csv) - источник:
[Kaggle: Walmart Recruiting - Store Sales Forecasting](https://www.kaggle.com/competitions/walmart-recruiting-store-sales-forecasting).

## Особенности данных

- Маркдауны (акции) заполняются нулями, а не средним - для большей части
  истории (до ноября 2011) акций такого типа просто не было
- CPI и Unemployment точечно пропущены по некоторым магазинам - заполняются
  ближайшим известным значением внутри магазина
- Lag- и rolling-фичи считаются отдельно для каждой пары Store-Dept, иначе
  модель будет подмешивать историю чужого отдела

## Метрика

WMAE (Weighted MAE) - официальная метрика конкурса: праздничные недели
(Super Bowl, Labor Day, Thanksgiving, Christmas) весят в 5 раз больше обычных,
потому что для ритейлера точность прогноза именно в эти недели критичнее всего.

## Результаты

| Модель             | MAE  | WMAE |
|---------------------|------|------|
| Linear Regression    | 1720 | 1851 |
| CatBoost              | 1395 | 1570 |

CatBoost получает Store и Dept напрямую как категориальные признаки и ловит
специфику конкретных магазинов/отделов, которую линейная регрессия в принципе
не может учесть без ручного onehot-кодирования на тысячи столбцов. Разница
хорошо видна в feature importance: лаги и WeekOfYear дают основной вклад,
но Dept тоже входит в топ-5.

## Запуск

```
pip install -r requirements.txt

```

## Что можно улучшить

- Кросс-валидация по нескольким временным окнам, а не один train/test сплит
- Отдельная модель или множитель на структурные нулевые продажи (закрытые отделы)
- Учёт удалённости от следующего праздника, а не только бинарный IsHoliday
